# ts-utils 0.8 Hardening Design

## Goal

Release `@maxxuxx/ts-utils@0.8.0` as a browser and server focused TypeScript utility package with safe defaults, deterministic edge-case behavior, reusable result and base64url primitives, and release verification that exercises the packed public API

## Release boundary

- Treat Electron removal and unsafe-contract cleanup as one pre-1.0 breaking release
- Keep the package root intentionally empty and preserve subpath-first imports
- Keep existing public names when their contracts can be made safe without ambiguity
- Remove caller-selected generic output types when no runtime validation supports them
- Do not restore Electron dependencies, exports, source, tests, or documentation
- Do not add signature verification to the JWT decoder
- Do not add deep object, general cookie, or separate date modules

## Architecture

### 1. Authentication and session safety

`api-fetch` validates the resolved request origin before attaching authentication

- The `baseURL` origin is trusted by default
- `allowedOrigins` explicitly extends the trusted set
- Requests to other origins remain possible but omit authentication headers
- Error contexts store a query-free and fragment-free URL
- Route responses expose mapped messages rather than raw upstream messages by default

Final authentication failure has one path

- Refresh throwing, returning no usable result, or producing another authentication response triggers best-effort session clearing
- The public failure is `ApiAuthError`
- The original failure is retained as `cause`
- Tokens, headers, bodies, query strings, and upstream response text are not copied into public messages

SvelteKit refresh separates shared refresh work from per-request persistence

```ts
type SvelteKitRefreshAuth<TCookies, TRefresh> = {
  namespace?: string;
  getAccessToken: (cookies: TCookies) => Promise<string | null | undefined>;
  getRefreshKey?: (
    cookies: TCookies,
    accessToken: string | null | undefined
  ) => Promise<string | null | undefined> | string | null | undefined;
  refresh: (
    cookies: TCookies,
    error: ApiHttpError
  ) => Promise<TRefresh | null | undefined>;
  applyRefresh: (
    cookies: TCookies,
    result: TRefresh
  ) => Promise<string | null | undefined>;
  clear?: (cookies: TCookies) => Promise<void>;
};
```

- Single-flight keys include namespace and stable refresh key
- The refresh operation runs once per key
- Every participating request calls `applyRefresh` with its own cookie context
- Failed refresh results are never cached

React sessions use memory persistence by default

- `storage: "memory"` is the default
- `storage: "local"` and `storage: "session"` are explicit opt-ins
- Restored and newly written sessions pass the configured session schema
- Corrupt persisted data is cleared instead of cast to the session type

### 2. API request lifecycle

The request pipeline has one ordered flow

1. Resolve URL and trusted origin
2. Parse query and request body inputs
3. Create a fresh raw body for the current attempt
4. Attach authentication only for an allowed origin
5. Compose source abort and per-attempt timeout signals
6. Execute fetch
7. Enforce response byte limit before parsing
8. Parse and validate the response
9. Normalize authentication, HTTP, parse, validation, abort, timeout, and size errors
10. Apply auth refresh once when eligible
11. Apply general retry policy when eligible

Public additions

```ts
type RawBodyFactory = (
  attempt: number
) => BodyInit | null | undefined | Promise<BodyInit | null | undefined>;

type RetryStrategy = "fixed" | "exponential";

type RetryOptions = {
  limit?: number;
  delay?: number;
  strategy?: RetryStrategy;
  jitter?: number;
  respectRetryAfter?: boolean;
};
```

- `rawBody` and `rawBodyFactory` are mutually exclusive
- `rawBodyFactory` is called for every network attempt including an auth retry
- `maxResponseBytes` rejects oversized bodies with `ApiResponseSizeError`
- Caller abort throws `ApiAbortError`
- Internal deadline expiry throws `ApiTimeoutError`
- Retry delays observe the caller signal
- `Retry-After` supports seconds and HTTP dates
- Jitter is a bounded ratio from `0` through `1`

### 3. Shared primitives

Add `@maxxuxx/ts-utils/result`

```ts
type Result<TData, TError = unknown> =
  | { ok: true; data: TData }
  | { ok: false; error: TError };

const ok = <TData>(data: TData): Result<TData, never>;
const err = <TError>(error: TError): Result<never, TError>;
const map = <TData, TNext, TError>(
  result: Result<TData, TError>,
  transform: (data: TData) => TNext
): Result<TNext, TError>;
const mapError = <TData, TError, TNextError>(
  result: Result<TData, TError>,
  transform: (error: TError) => TNextError
): Result<TData, TNextError>;
```

- `JsonResult`, `JwtResult`, `EncodingResult`, and promise result aliases reuse `Result`
- Existing result object shape remains unchanged
- Module-local duplicate success and failure factories are removed

Add `@maxxuxx/ts-utils/encoding/base64url`

```ts
const encodeBase64Url = (value: string | Uint8Array) => string;
const decodeBase64Url = (value: string) => Uint8Array;
const decodeBase64UrlText = (value: string) => string;
const isBase64Url = (value: string) => boolean;
```

- Accept only the base64url alphabet with valid optional terminal padding
- Reject impossible lengths and non-canonical padding
- Use fatal UTF-8 decoding for text
- Produce the same result in Node and browser runtimes
- JWT delegates segment decoding to this module

Add `createSingleFlight` to the promise module

```ts
type SingleFlight<TKey, TValue> = {
  run: (key: TKey, task: () => Promise<TValue>) => Promise<TValue>;
  clear: (key?: TKey) => void;
  readonly size: number;
};

const createSingleFlight = <TKey, TValue>(options?: {
  successTtlMs?: number;
}) => SingleFlight<TKey, TValue>;
```

- In-flight work is shared per key
- Rejections are removed immediately
- Successful values are retained only for the configured non-negative TTL
- Session and SvelteKit refresh reuse this primitive instead of separate global maps

### 4. Runtime correctness

Object helpers create own data properties for every key including `__proto__`

- `pick`, `removeUndefined`, `compact`, and `fromEntries` never invoke prototype setters
- The result retains ordinary object semantics

JSON helpers distinguish shared references from cycles

- The recursion stack removes a value after its branch completes
- Shared children are accepted
- True circular structures are rejected
- `safeParseJson` returns `Result<unknown, JsonParseError>`
- Typed parsing requires `safeParseJsonWithSchema`

JWT helpers use strict base64url and schema-backed typing

- `decodeJwt` returns `JwtPayloadWithToken | null`
- `safeDecodeJwt` returns `JwtResult<JwtPayloadWithToken, JwtDecodeError>`
- `decodeJwtWithSchema` and `safeDecodeJwtWithSchema` provide typed claims
- Header functions follow the same rule

Date and time helpers validate the final numeric result

- Invalid `Date` values use the documented fallback or fail explicitly
- Clock snapshots reject non-finite results
- Server round-trip samples reject inconsistent timestamp ordering and negative round-trip time
- Server-time payload creation rejects values outside the JavaScript Date range

Promise helpers validate platform timer bounds

- Timer delays must be between `0` and `2_147_483_647`
- `sleep` accepts an optional abort signal
- Timed tasks receive an `AbortSignal`
- Timeout aborts the attempt before retry begins
- Documentation states that cancellation depends on the task observing the signal

Environment getters parse injected raw values without stringifying first

- Unsafe numeric bigint values are rejected
- Getter and schema boolean semantics match
- String getters remain text based

Browser device cookies reject `SameSite=None` with `secure: false`

Other correctness changes

- `isPrimitive(NaN)` returns true
- `isFalsy(0n)` matches its declared type
- Parser numeric presets reject booleans
- Korean phone formatting validates supported prefixes
- URL query appending preserves the complete fragment after the first `#`
- `DeviceUuidParseError` is exported from Node and aggregate device entries
- Common HTTP response helpers add forbidden, not found, conflict, unprocessable entity, and no-content responses

## Error model

- Safe APIs return `Result`
- Throwing APIs use a module-specific error with `cause`
- Boundary validation failures never silently coerce unsupported values
- Public error messages exclude tokens, request headers, request and response bodies, query strings, fragments, and raw upstream text
- Retry applies only to explicitly eligible failures and replayable request bodies
- Caller abort, timeout, response size, auth, HTTP, parse, and schema failures remain distinguishable

## Testing strategy

- Every behavior change starts with one focused failing regression test
- Confirm each test fails for the expected pre-change behavior before implementation
- Run the owning module test after each minimal fix
- Run cross-module tests after shared Result, base64url, single-flight, session, and api-fetch changes
- Add type assertions for removed unchecked generics and schema-backed replacements
- Exercise Node and browser base64url paths
- Exercise concurrent SvelteKit requests with distinct cookie contexts
- Exercise auth omission on cross-origin requests and auth inclusion on explicitly allowed origins
- Exercise abort during delay, timeout cancellation, `Retry-After`, jitter bounds, body recreation, and response limits
- Run typecheck, all tests, clean build, export self-import, dry pack, dependency audit, and Electron absence checks before completion

## Migration policy

- Breaking changes are documented with old and new usage
- Deprecated compatibility aliases remain only where they preserve the new safe runtime behavior
- No compatibility shim may restore unsafe localStorage defaults, unchecked typed decoding, cross-origin auth forwarding, raw error disclosure, or context-unsafe refresh dedupe
